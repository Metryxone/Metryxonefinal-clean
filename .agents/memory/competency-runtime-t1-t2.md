# T3/T4 — question→competency mapping & precise scoring upgrade
- The runtime scorer has TWO modes: domain-`PROXY` (7 bank codes crosswalk DOWN to 5 onto-domains — the legacy default) and `precise` (per-competency, driven by `onto_competency_question_map`). `scoreAssessment` emits `measurement: 'domain_proxy' | 'precise' | 'hybrid'` + a `competency_scores[]` array; empty map → `domain_proxy` + `competency_scores: []` = byte-identical legacy.
- **Auto-upgrade is data-driven, not a flag**: mapping a question (writing an active `onto_competency_question_map` row) is the ONLY action needed; the next score of any instance whose questions include mapped ones upgrades that competency to `precise`. Only the mapped questions PRESENT IN THE INSTANCE contribute — so a blueprint can show 2 precise competencies even if 7 are mapped (honest, not a bug).
- **Reuse the existing writer**: `bulkMapCompetencyQuestions(pool,{pairs,source})` in `services/assessment-foundation-mapping.ts` does normalise+dedupe, validates BOTH comp (onto_competencies.id) and question (uuid) existence, returns a `skipped[]` ledger with reasons (`competency_not_found`/`question_not_found`) — surface it, never silently drop. Routes: GET `/mapping-grid`, GET `/competency-map`, POST `/competency-map/bulk` (LITERAL before any `/:param`).
- **T4 seed already existed** — don't write a new one: `runAssessmentFoundationSeed(pool)` (`scripts/seed-assessment-foundation-mapping.ts`) calls `deriveCompetencyQuestionMap`, which resolves each template's OWN `competency_code` → competency by id-then-`canonical_name`, inserts `source='derived'`, and honestly skips unresolvable codes. In dev that yields 23 derived rows / 7 competencies; the 21 legacy SHORT codes (COM/EXE/LEA/TEC/EIQ/COG/ADP) are NOT onto ids and stay unresolved — mapping them would be fabrication.
- **API envelope double-wrap trap**: `/assessment-instances` (generate) returns the handler object `{ok,instance_id,...}` AND `wrap()` adds an outer `{ok,version,data}` → fields live at `data.instance_id`, `data.total_questions`, `data.questions`. The grid endpoint nests at `data.questions/competencies/total_*`.
- `onto_competencies` name column is `canonical_name` (NOT `name`); `onto_competency_question_map` has UNIQUE(competency_id, question_id) and the bulk upsert uses `(xmax=0)` to split inserted vs reactivated.

# T5 — adaptive / branching runtime
- Adaptive runtime is a PURE re-ordering of the already-generated instance pool — NO new question generation, NO new persistence table. `submitSingleResponse` upserts ONE row into `onto_assessment_responses` (ON CONFLICT (instance_id, question_index)); `nextAdaptiveQuestion` reads responses, calls the pure `chooseBranch` (services/adaptive-branching-engine.ts), realizes the decision against `remaining` items, and degrades to next-sequential (`branch.fallback=true`) inside a try/catch so the assessment ALWAYS completes. `scoreAssessment` now falls back to persisted responses when `input.responses` is empty → incremental + batch share one score path.
- **Do NOT persist to `adaptive_question_branches`**: that table is FK-bound to `dynamic_question_sessions`; a competency-runtime instance is NOT a session row, so writing it requires fabricating a session (the FK error `adaptive_question_branches_session_id_fkey` is the tell). Surface the branch decision in the `/next` response (`branch` field) instead — honest provenance, zero fabrication.
- **PRE-EXISTING 404-body quirk (NOT a T5 bug)**: error paths do `res.status(404).json({ok:false,error})` but the client receives the generic wrap envelope `{ok:true,version:"phase-2"}` with a 404 status. The already-reviewed `/score` route behaves IDENTICALLY, so it's established `wrap`-helper behavior across this whole file, not a regression. `backend/index.ts:84` only captures res.json for logging (pass-through). Status codes are correct; don't "fix" it inside a T5 scope.
- **Backend API workflow flakiness**: restart_workflow reports `failed` (port-detection false-fail per backend-workflow-port-detection.md) and leaves the server intermittently unreachable / orphan processes that serve stale responses. ALWAYS poll `/api/competency-runtime/blueprints` for 401/200 before trusting a smoke test; transient `000` (connection refused) = mid-restart, not a code defect.

# Coverage-vs-Confidence surfaced to end-users (reliability block)
- The end-user score endpoint emits an additive, flag-gated `reliability` block that reports TWO independent axes that must NEVER be merged into one number: **Coverage** (competencies_scored / 20, domains_covered / 7 — i.e. *does the data exist*) and **Confidence** (mean captured response-confidence, banded high≥0.8 / moderate≥0.6 / low, else `unmeasured` — i.e. *is it trustworthy*). Empty path is `0/20 + unmeasured`, NEVER `0` (an absent axis is not a zero score).
- **Why:** users misread a single composite as "I scored low"; separating the axes lets an honest "we don't have enough data yet" read as coverage, not failure. Confidence is *captured self-report confidence*, NOT external/psychometric validation — the basis note must say so or it overclaims.
- **How to apply:** gate the whole block behind the runtime flag so flag-OFF omits the field (byte-identical legacy payload); render both axes with an explicit separation caption in the UI.

# CAPADEX behavioural signals → behavioural competency dimension (concern-diagnostic lowering)
- CAPADEX behaviour-graph risks are **concern-DIAGNOSTIC**: when fed into the behavioural competency dimension (`dom_behavioral`) they may only **LOWER or FLAG** it as risk — they may NEVER raise it as a strength, and they sit at/below a NEUTRAL ceiling (50). The evidence object starts at a neutral base and is only dragged DOWN by severity-weighted deficits; no risks → `evidence_score: null` (unmeasured), never a fabricated 0 or neutral.
- The dimension feed is **downward-only and non-mutating**: surface an `effective_score = max(0, base - totalRiskDeficit)` alongside the untouched `base_score`, with `applied=true` ONLY when a measured base AND risk evidence both exist; otherwise `effective_score == base_score` (unchanged). Persisted profile/domain scores are NEVER written.
- **Identity bridge is mandatory** and reuses the existing CAPADEX behaviour bridge (`capadex_behavioural_memory.user_id → session_id → capadex_behavior_graph` via `buildBehaviorGraphForUser`); no linked session → `available:false` honest path, never invent a graph.
- **Why:** the platform-wide strengths canon (signals are concern-diagnostic, strengths come ONLY from CSI positive_factors) must hold even when behaviour feeds a *competency* score — an uplift here would silently turn a concern signal into a strength. Surfacing base+effective separately keeps the measured proxy auditable while still applying the risk.
- **How to apply:** compose the evidence service (never recompute the graph); two consumers — the CEI summary endpoint (`behavioural_evidence`) and the runtime dashboard (`behavioural_dimension`), each conditional-spread behind the runtime flag (flag-OFF omits the key = byte-identical) and never-throws (degrade to null/omitted, leaving prior behaviour intact). Seeding the applied path in dev needs a real `onto_assessment_instances` row (profile.instance_id is a NOT-NULL uuid FK) — reuse an existing instance id.

# Reverse-scored (negative-polarity) questions
- Polarity lives in `template_body.reverse_scored` (JSONB, additive — NO column/migration needed).
- Inversion happens in `deriveOptions` (backend/services/competency-runtime.ts): Likert scores invert across the 0..100 scale (`100 - score`) so "Strongly Agree" on a negative item scores LOW. **Only Likert** — best-answer types (mcq/sjt/scenario/case/simulation/behavioral/communication) have a single correct answer, polarity is meaningless there, flag ignored → legacy byte-identical.
- The inverted scores are baked into the generated instance `questions` JSON at generate-time, so the score path (`scoreAssessment`) stays unchanged. RuntimeQuestion carries `reverse_scored?:true` for traceability (set only when Likert AND flagged).
- **Why:** keeps scoring path byte-identical; a question with no flag derives identical options to before.
