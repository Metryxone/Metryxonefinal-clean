# Enterprise Intelligence Integration Platform (MX-800 Phase 2.13)

Flag `enterpriseIntelligenceIntegration` / `FF_ENTERPRISE_INTELLIGENCE_INTEGRATION` (default OFF). ENHANCEMENT-ONLY read-only composer over the EXISTING tiers; OFF byte-identical incl. schema. NO new engine / V2 / parallel / duplicate registry; NO business-logic change; NO dormant activation.

## Durable lessons
- **It COMPOSES read-only SUMMARY getters — it does NOT "never invoke engines" like 2.12.** `composeServices` actually calls `get*Summary` / `getAutomationGovernanceCatalog` / `buildWorkflowOverview`. The safety claim is `invokes_engine:false` meaning it never runs an engine's compute/write/scheduler/emit/execute path — the summary getters it calls are themselves read-only composers. When writing a validator, assert *positive* read-only-summary imports + *negative* emit/scheduler/`.on()` — do NOT copy 2.12's "engines-never-invoked" assertion verbatim.
- **`composeServices` is memoized once per process (memo `eii:composition`); all heavy getters reuse ONE composition pass.** Unlike 2.12 (no shared substrate memo, each getter re-scans), here cross/composition/coordination/validation/metrics/summary all share the single pass — so they can run in one phase. Still phase-split for clean baselines + the 2-snapshot drift split (write leaves tables, drift reuses + cleans up).
- **Catalog is 14 services, not the plan's 15.** Measure the real `INTEGRATION_SERVICES` array (`uid:`/`name:` count), never trust a plan estimate. Distribution: 9 intelligence + 1 enterprise(2.12) + 1 automation(1.41) + 2 platform(1.37 foundation, 2.11 ops) + 1 reporting. Catalog totals key is `services` (not `capabilities` like 2.12).
- **11 of 14 are getter-backed; 3 are HONEST non-getters** (1.37 foundation, 2.11 ops center, report-factory) registered by existence with `reachable:null` (null ≠ 0, NOT a failure). 9 of the 11 are the intelligence channels → `cross.reachability.of === 9`; `composition.totals.getter_backed === 11`.
- **6 SEPARATE metrics, `composite:null`, `enterprise_readiness` honest-null (DEFERRED).** Operational readiness needs runtime+outcome evidence (does the integrated platform actually operate as one) → absent → null. Integrated ≠ Unified; Unified ≠ Operational. Axes span Structural ⟂ Confidence ⟂ Coverage ⟂ Outcome — never composited. `workflow_health` is STRUCTURAL reachability (100 reachable / 50 degraded / null unreachable), NOT runtime throughput.
- **2 owned tables only** (`enterprise_integration_registry` + `enterprise_integration_audit_snapshots`). Writes `assertEnabled()` THEN `ensureSchema()` → OFF creates 0 tables; reads `to_regclass`-probe (registry/snapshots/drift → `present:false` when absent). `/audit/capture` is the ONLY write path.
- **Registry rows are raw DB columns** — the FE panel Registry tab keys must match the migration columns (`name`/`service_kind`/`tier`/`present`/`table_count`/`flag_state`/`lifecycle_state`/`owner`). `lifecycle_state`+`owner` are MANAGED (human, preserved across re-discover); `present`/`table_count`/`flag_state` are DERIVED (refreshed on re-discover).
- **Injection guard:** `POST /register` `physical_table` is rejected by `isSafeTableIdentifier()` before any interpolation (exported for the validator).
- **FE Part 9 wiring = 4 spots in `SuperAdminDashboard.tsx`** (lazy import + `/feature-flag` `res.ok` useQuery gated on `isAuthenticated` + extraTabs conditional-spread + top-level `activeTab` block); tab hidden OFF. Validate the frontend via esbuild parse + dist grep, NEVER vite build / pkill (pathologically slow + pkill kills the shell).
- **Validator** `scripts/mx800-2.13-integration-validate.ts` (phased `light,compose,metrics,summary,write,drift`; 42/42). Sets `FF_ENTERPRISE_INTELLIGENCE_INTEGRATION=1` + `EII_MEMO_TTL_MS=300000` at top; drops the 2 owned tables start+end (`skipStartDrop`/`skipEndDrop` for the write→drift continuation).
