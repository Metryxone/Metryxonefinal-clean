# WC-C3 — Entitlement ENFORCEMENT audit

Follow-on to WC-C2. Measures whether *payment controls access* (entitlement) and *role controls admin* (RBAC), as a live static-analysis + DB audit. The deliverable IS the audit (`backend/audit/wc-c3/` — 6 markdown + `_wc_c3_snapshot.json`), generated by a script that re-derives everything live.

## Central honesty trap (carries forward from WC-C2 — do not regress)
- The objective's premise "20% → >90% **through enforcement**" is **FALSE** on the named *Product Monetization Readiness* (WC-C1 6×5) metric. Enforcement only moves that metric to ~20%; reaching >90% there is **productization** (building more SKUs), not enforcement.
- Enforcement reaches ≥90% **only** on a *re-baselined Live-SKU* denominator (CAPADEX ladder is the sole real consumer SKU). Present that re-baseline as a **user DECISION**, never silent-swap the denominator.
- **Why:** the named metric is reserved to its original denominator; swapping it to flatter a verdict is the exact inflation this audit family must avoid.
- **How to apply:** answer on the named metric FIRST and state the premise is false there, THEN offer the Live-SKU path as an explicit decision.

## Measurement traps
- **Guard enumeration must credit THREE forms** or you over-report the admin gap: (1) inline middleware in the route def, (2) spread arrays (`...adminChain` / `...adminGuards`), (3) global `app.use('/api/admin/<prefix>', guard)` prefixes. A naive inline-only grep counted ~340 unguarded; crediting spread+global brought the honest number to **212/623**.
- The 212 figure is **"no route-level guard DETECTED (static analysis; in-handler checks not inspected except a confirmed sample)"** — NEVER "unprotected/exploitable." Disclose the static-analysis limitation **wherever the number appears** (incl. the executive summary).
- **Quarantine RBAC from entitlement.** Admin-RBAC is a SEPARATE security axis; compositing it into the entitlement % would let a security sweep inflate the monetization verdict (or vice-versa). Keep the entitlement readiness cells entitlement-only.
- `entitlement_guard_exists` is genuinely **false** here — zero `requireEntitlement`/tier guards in the whole codebase. Paid-tier CAPADEX endpoints are served on **session-UUID possession = bearer token** (no backend paywall).
- Re-derive the paid-endpoint set fresh each run from `routes/capadex.ts` (report/:session_id + /pdf + /send-email POST + session/:id/{explain,guidance,grounding,omega-x,patterns,pipeline,stage,report,reports,signals}). Some are not even flag-gated — report `runtime_flag` honestly per endpoint; checking only the activation flag *over*states exposure (conservative, acceptable, but note it).

## Dual-axis + projection rules
- **Structural** (code exists) vs **Activation** (live + real paid volume) are NEVER composited into one number.
- Post-keystone projection caps flag-gated cells at **gated-real(4)**, never real(5), while `commercialEntitlement` defaults OFF; **Activation stays 0%** ("cannot be raised by configuration").
- `enforcement_middleware` + `report_endpoint_protection` are **dependent** — one `requireEntitlement` build flips BOTH cells; do not count as two independent wins.

## Effort estimate placement
- Snapshot JSON carries **countable units + T-shirt size only**. The engineering-**day** band lives **only** in the roadmap prose under a "judgment, not measurement" header. Grep the snapshot to confirm zero day figures.
- **Why:** day numbers in a machine-readable artifact read as fabricated precision; the countable units are the measurable part.

## Verdict discipline
- Run the script **without** `FF_*` env overrides so `FEATURE_FLAGS` reflects deploy-posture defaults (`commercialEntitlement`/`commercialActivation` = false).
- Architect honesty review (evaluate_task) is mandatory; treat its invariant checklist as the gate. PII sha256-masked everywhere.
