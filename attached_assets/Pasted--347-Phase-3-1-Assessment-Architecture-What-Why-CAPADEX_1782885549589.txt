#347 - Phase 3.1 Assessment Architecture
What & Why
CAPADEX 3.0 · Program 3 · Phase 3.1 — Assessment Architecture · Lifecycle · Governance. Precondition met: Phase 3.0 Assessment Platform Blueprint is APPROVED and frozen.

Establish and certify the ONE canonical Assessment Architecture defined by the approved blueprint — enhancement-only, reuse-before-build, no duplicate frameworks, no breaking changes. The mandatory discovery pass (already run) proves most of this architecture already exists as pure-data registries; this phase consolidates it into one coherent, certified architecture layer and produces 10 evidence-grounded deliverable docs, then STOPS for human approval before Phase 3.2 (Question Management Platform).

Execution contract (from the spec, non-negotiable): Repository-first · Blueprint-first · Enhancement-only · Search-before-modify · Reuse-before-build · Preserve existing architecture · No duplicate assessment frameworks · No breaking changes · Additive/flag-gated/byte-identical-off · Human approval required.

What ALREADY exists (from discovery — reuse, do NOT rebuild):

Canonical type taxonomy: FROZEN 10-type registry in backend/config/assessment-framework.ts (entry/baseline/diagnostic/behaviour/competency/learning/performance/progress/exit/continuous) + SPEC_19_CROSSWALK. Mappings: assessment-framework.ts (type→persona/lifecycle/journey/competency/scoring/benchmark/AI/rec/report/outcome/KPI), customer-journey.ts, ai-orchestration-model.ts; traceability composers scripts/capadex-1.3-*/1.4-*; blueprint docs 08/17.
Honest architecture gaps the phase must name (not silently paper over):

Spec lists 15 canonical TYPES; repo has 10 (frozen). Aptitude / Organization / Custom are NOT first-class named types — must be adjudicated via crosswalk (fold-in vs honest-absent), never fabricated.
No dedicated assessment CATEGORY registry — categories exist only as industry tags (IND_EDU/IND_GOV/IND_HEALTH/IND_CORP), roles, and behavioural domains. Government/Healthcare/NGO/Research categories are non-validated scaffolds (per Phase 3.0 boundary markers — no clinical/diagnostic claims).
No single assessment-LEVEL 10-state lifecycle (Draft→Review→Pilot→Validation→Approval→Published→Active→Suspended→Deprecated→Archived) — existing lifecycles are per-question / per-content-draft / per-capability. This phase encodes ONE canonical lifecycle MODEL that composes/maps onto the existing per-artifact machinery (registry/model), it does NOT rip out or replace them.
No single canonical metadata STANDARD (the 18 fields: id/version/owner/business_domain/persona/lifecycle_stage/competencies/behaviours/skills/duration/language/difficulty/type/category/tags/status/created_by/updated_by) applied uniformly — metadata is rich but per-registry/per-table.
Done looks like
ONE canonical assessment architecture, lifecycle model, governance model, metadata standard, and mapping model — encoded as pure-data registries + a read-only composer/verifier + flag-gated routes (mirroring the CAPADEX 1.3–1.7 pattern), default OFF and byte-identical-off including schema (zero DDL / reads-only).

Final verdict answers: "Is the Assessment Architecture complete and ready for Phase 3.2?" with every remaining gap classified Launch-Critical / High / Medium / Low / Future, citing REAL repo evidence (MISSING = MISSING).
Existing APIs / DB / frontend unchanged and verified aligned; no duplicate framework introduced.
STOP for human approval — nothing proceeds to Phase 3.2 until sign-off.
