## Canonical realization loop (TARGET — the close-the-loop spine)
`Baseline → Act/Grow → Re-measure (Progress) → Exit assessment → Realized outcome capture → Attribution →
Outcome KPI (14).` Today the loop is **front-loaded**: it can *project* and *abstain* but cannot yet *measure
realized outcomes* because there is no live adoption data pre-launch (abstention < k_min=30 is correct, not a
defect).

## Why this is the keystone gap
An enterprise buyer asks "does it change outcomes?" The honest answer today: **"we measure inputs and project
outcomes; realized-outcome evidence is not yet captured."** This is why **production-confidence is WITHHELD
(null)** across the entire program — by design, not by failure. Outcomes ≠ reports.

## Canonical decisions (FROZEN)
1. The 6-type outcome taxonomy is the **canonical target**; do **not** mutate the live validation-loop
   `OUTCOME_TYPES` to match — keep this as the product-level taxonomy.
2. **Coverage ⟂ Confidence never composited; null ≠ 0.** Demo data is excluded from outcome coverage AND ledger
   in lockstep (or coverage self-inflates).
3. Realized-outcome capture + recommendation-effectiveness feedback is the **Tier-1 enhancement** (GAP-O1) — the
   one that converts **Managed → Intelligent** maturity post-launch.
