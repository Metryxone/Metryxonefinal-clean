# Context & Objective
We need to fix a critical disconnect in the CAPADEX engine where the final Clarity Report (`GET /api/capadex/report/:session_id`) completely ignores behavioral telemetry (BIOS signals captured via `POST /api/signals/ingest` and saved to `capadex_signal_profiles`/`capadex_linguistic_signals`). The report feels generic because it relies 100% on explicit questionnaire responses, ignoring implicit markers like hesitation, response volatility, linguistic load, or social desirability bias.

Refactor the report compilation pipeline across the backend and frontend to fully synthesize explicit performance scores with implicit behavioral signals into an integrated intelligence layer.

---

## Constraints (Do Not Violate)
1. **Append-Only / Read-Only Safety**: Do NOT change or run migrations on the underlying Drizzle PostgreSQL schema definitions. Use the existing `capadex_sessions`, `capadex_reports`, `capadex_signal_profiles`, and `capadex_linguistic_signals` configurations.
2. **Deterministic UI Fallbacks**: If a session lacks a corresponding BIOS profile record (e.g. tracking failed or telemetry was blocked), the code must gracefully default to displaying explicit content metrics without breaking or throwing `500`/TypeError states.
3. **Language Policy**: Synthesized narrative blocks must serve exclusively as *developmental signals only*. Do not emit any text implying diagnostic profiling, automated corporate hiring suitability, or clinical evaluations.
4. **k-Anonymity Compliance**: Any peer benchmarking components or aggregate statistics surfaced in report visual nodes must instantly suppress metrics if the active cohort baseline density drops below $k_{\min} = 30$.

## Technical Specifications & Implementations Required

### Step 1: Backend Route Data Unification
File: `backend/routes/capadex.ts`
Locate the route handling `GET /api/capadex/report/:session_id`.
1. Update the database lookup sequence using Drizzle ORM queries to fetch the following matching entities using the target `session_id`:
   - `capadex_reports` & `capadex_sessions`
   - `capadex_signal_profiles`
   - `capadex_linguistic_signals`
2. Construct a synthesized output payload dictionary combining explicit metrics with an optional behavioral analysis envelope (`behavioral_signals` and `linguistic_context`).

### Step 2: Synthesis Engine Parser Injection
File: `backend/routes/capadex.ts` (or a dedicated service file called by it)
Implement a pure synthesis evaluation function to generate a behavioral archetype payload. Match these concrete combinations:
- **High Score ($\ge 80$) + High Load/Hesitation (`cognitive_load` or `emotional_load` $> 70$)**: Append archetype metadata indicating an `Over-Indexed Optimization Strain`. Text should reflect that correct responses are being powered by high cognitive friction or overthinking patterns.
- **High Score ($\ge 85$) + Ultra-Low Response Pace (`rapid_answer` pattern triggered / average time $< 800\text{ms}$)**: Append archetype metadata indicating a `Potential Social Desirability Bias`. Text should flag a potential automatic "ceiling effect" selection style.
- **Low Score ($< 50$) + High Cognitive Stability (Low Volatility / low variance)**: Append archetype metadata indicating an `Authentic Structural Growth Matrix`. Text should emphasize high baseline calibration ready for immediate skill guidance.

Ensure this synthesized data is securely appended to the response JSON payload.

### Step 3: Frontend Report Dashboard Enrichment
File: `frontend/src/components/assessment/phases/CapadexReportPhase.tsx` (or your matching platform UI component parsing the report route payload)
1. Safely extract the new behavioral envelope inside your state interface.
2. Build a clean, visually scannable **Behavioral Intelligence Overview Grid** adjacent to the subdomain bars. It must display 4 simple gauge bars or micro-progress rungs:
   - *Emotional Burden Index* (`emotional_load`)
   - *Cognitive Processing Pace* (`cognitive_load`)
   - *Engagement Fidelity* (`engagement_score`)
   - *Linguistic Precision Factor* (`absolutism_score` derived from the text analysis frame)
3. Use your design system color tokens (`Tokens.ts`) to drive the alert statuses:
   - Load metrics $\le 40$: Green (Stable Environment Context)
   - Load metrics $41 \dots 70$: Orange (Moderate Adaptation)
   - Load metrics $> 70$: Red (High Cognitive Friction)

### Step 4: Outbound Branded Email Synchronization
File: `backend/email.ts`
1. Update the HTML generator function mapped to `POST /api/capadex/report/:session_id/send-email` to accept the integrated profile metrics.
2. Inject a clean structural summary text block reflecting the synthesized archetype directly into the email body text.
3. Preserve the current strict ASCII protection rule: the `X-Preview-Subject` header **MUST** be processed via `encodeURIComponent(subject)` to prevent em-dash (`—`) characters from crashing the node runtime transport pipeline.
